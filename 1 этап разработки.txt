# Документация проекта: Merchberry ERP

## 1. Концепция и видение продукта (Vision)
**Merchberry ERP** — это легковесная, быстрая и защищенная система управления предприятием (ERP), разработанная специально для учета товаров, финансов и взаимоотношений с арендаторами полок/авторами мерча. 

Система решает главную боль розничных точек: автоматизирует расчет комиссий, избавляет от потери данных при обрывах связи благодаря системе "Буферной зоны" для товаров и предоставляет удобные ролевые интерфейсы для Администратора и Продавцов.

---

## 2. Технологический стек
**Архитектура:** Клиент-серверная (SPA + REST API)

**Backend (Серверная часть):**
* Язык: Python 3.12+
* Фреймворк: FastAPI
* База данных: SQLite (через ORM SQLModel)
* Авторизация: JWT-токены, хэширование паролей (чистый `bcrypt`)
* Фоновые задачи: APScheduler (планировщик)
* Обработка файлов: встроенный парсер CSV/Excel (с поддержкой разных кодировок и разделителей)

**Frontend (Клиентская часть):**
* Сборщик: Vite
* Библиотека: React (TypeScript)
* Стилизация: Tailwind CSS v4
* Маршрутизация: React Router DOM v6
* HTTP-клиент: Axios (с настроенными Interceptors для токенов)
* Иконки: Lucide React
* Графики: Recharts

## 3. Реестр выполненных задач (Что уже готово)

### 3.1. Базы данных и Модели (`models.py`)
* Спроектирована таблица `User` с разделением ролей (`admin`, `seller`), учетом финансовых реквизитов, процента комиссии и номеров полок.
* Спроектирована таблица `Product` с поддержкой "Ничьих" товаров (`seller_id = None`) для безопасного импорта.
* Спроектирована таблица `Transaction` для ведения единой финансовой книги (продажи, аренды, выплаты, ручные корректировки).

### 3.2. Авторизация и Безопасность (`auth.py`, `init_admin.py`)
* Реализована строгая система ролей. Созданы функции-охранники (`get_current_user`, `get_current_admin`), защищающие эндпоинты.
* Настроена генерация и валидация JWT-токенов.
* Написан seed-скрипт `init_admin.py` для первичной инициализации базы данных и создания суперпользователя.

### 3.3. API и Бизнес-логика (`main.py`)
* **Парсер CSV:** Разработан мощный парсер, который игнорирует невидимые символы Excel (BOM), автоматически определяет разделитель (запятая или точка с запятой) и ищет нужные колонки по корням слов, игнорируя регистр.
* **Импорт в 2 шага:** Эндпоинты `/preview` (анализ файла) и `/confirm` (сохранение в БД) для защиты от потери данных.
* **Графики:** Эндпоинт `/analytics/chart` с серверной агрегацией (разбивка продаж по дням или неделям в зависимости от выбранного периода).
* **CORS:** Настроено безопасное взаимодействие портов `8000` и `5173`.

### 3.4. Интерфейс (Фронтенд)
* **Инфраструктура:** Настроен единый API-клиент (`axios.ts`), который сам прокидывает токены и обрабатывает потерю сессии (выброс на `/login`).
* **Оболочка (`Layout.tsx`):** Боковое меню с динамической отрисовкой пунктов в зависимости от роли (Админ видит больше, чем Продавец).
* **Дашборд (`Dashboard.tsx`):** Страница с виджетами и интерактивным графиком продаж. При клике на столбец графика появляется детализация проданных товаров за этот период.
* **Склад (`Inventory.tsx`):** Таблица товаров с цветовой индикацией остатков. Форма загрузки CSV с предпросмотром добавленных/измененных позиций. Возможность Админа привязывать "ничейные" товары к конкретным продавцам в два клика.

## 4. Задачи в процессе (Ближайший фокус)

**Проблема:** Механизм передачи CSV файла от Axios к FastAPI. 
**Статус:** Логика на бэкенде полностью написана, но на стороне браузера (фронтенда) происходит сбой заголовков (`multipart/form-data`). 
**План решения:** Написать изолированный тестовый запрос через `fetch` (вместо `axios`), чтобы точно убедиться, как именно браузер формирует "границы" (boundary) файла, и заставить FastAPI корректно его прочитать.

## 5. Дорожная карта (Roadmap) и плановые рекомендации

### Этап 1: Завершение складского учета (Inventory)
* **Задача:** Успешно прокинуть CSV-файл через фронтенд в БД.
* **Рекомендация:** Отказаться на 15 минут от React и отправить файл через чистый HTML/Postman. Как только убедимся, что бэкенд его "съел", перенесем точную копию запроса в React.

### Этап 2: Управление Продавцами (Users/Sellers)
* **Задача:** Создать интерфейс для Админа (создание авторов, назначение им комиссии, указание номеров полок).
* **Рекомендация:** Сделать простую CRUD-таблицу. Форма создания продавца должна автоматически генерировать ему пароль и показывать его админу.

### Этап 3: Финансовая сводка (Finance)
* **Задача:** Отрисовка таблицы `Transaction`.
* **Рекомендация:** Для продавца это будет просто история "read-only" (поступления с продаж). Для Админа нужно добавить кнопку "Создать транзакцию" (например, списать аренду с баланса или зафиксировать выплату на карту).

### Этап 4: Робот синхронизации (Интеграция с Бизнес.Ру)
* **Задача:** Оживить фоновую задачу `sync_sales_from_business_ru` и кнопку ручной синхронизации.
* **Рекомендация:** Как только будут получены `app_id` и `secret`, реализовать функцию формирования MD5-подписи (скелет уже заложен). Интеграцию лучше тестировать на одном конкретном историческом дне, в котором точно были продажи.

### Этап 5: Деплой (Вывод в Production)
* **Задача:** Перенос системы с локального компьютера на боевой сервер.
* **Рекомендация:** 1. Завернуть Фронт и Бэк в Docker-контейнеры.
  2. Заменить SQLite на PostgreSQL для надежности при одновременных запросах.
  3. Настроить Nginx как реверс-прокси и прикрутить SSL-сертификаты.
